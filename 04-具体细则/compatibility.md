## [Must] 保持向后兼容性
我们可以更改 API，但要确保所有使用者正常运行。API 消费者通常拥有独立的发布周期，注定稳定性，避免不提供额外价值的更改。API 是服务提供者和服务消费者之间的合同，不能通过单方面的决定来破坏。

有两种技术可以在不破坏 API 的情况下改变 API：
- 遵循兼容扩展的规则
- 引入新的 API 版本，并支持旧版本

我们强烈建议使用兼容的API扩展，不鼓励版本化。
